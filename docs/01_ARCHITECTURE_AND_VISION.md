# 1. 架构与愿景：降维执行矩阵与物理隔离

## 1.1 工程背景与降维重设计
传统操作系统在解决安全与性能矛盾时，陷入了基于 Ring 0/3 的设计瓶颈。软件故障隔离（SFI）引入高昂的指令级边界检查开销；系统调用（Syscall）特权级切换在超标量 CPU 上导致严重的流水线冲刷延迟；而与内核同权的第三方驱动则构成了永恒的稳定性黑洞。

Aether Hyperion 架构的核心结论是：**不在软件层面解决硬件隔离问题**。本系统将自身特权级降维至硬件虚拟化扩展层（Intel VT-x / AMD-V），构建一个纯粹的执行矩阵，在保持 100% 原生 ALU 吞吐量的同时，实现物理强制隔离。

## 1.2 核心系统架构
架构由极简的底层调度器与多种形态的并行执行域构成，两者存在绝对的物理隔离边界。

### The Hypercore (Ring -1)
系统的绝对基底，使用无 `unsafe` 的 Rust 核心子集编写，执行编译期验证，并为形式化验证做准备。
*   **职责**：CPU 周期分配、物理页框（EPT）路由、IOMMU 硬件配置、外部中断再分配。
*   **约束**：内部不包含文件系统、网络栈或任何传统设备驱动。

### 多态隔离域 (Polymorphic Enclaves)
所有逻辑单元均运行于 VMX Non-Root 模式。
1.  **极速微域 (Micro-Unikernel)**：直接运行于 Guest Ring 0。应用与必要运行时静态链接。微秒级启动，提供确定性延迟。
2.  **宏域 (Macro-POSIX Realm)**：内部恢复 Ring 0/3 划分，运行兼容性系统（如 Linux 宏内核），无损承载传统应用与桌面生态。
3.  **闭源外设域 (Blackbox Driver Realm)**：为闭源二进制（如 GPU 驱动）分配独立域。通过 IOMMU 直通物理寄存器，驱动错误仅导致单域崩溃，且内存被物理阻断。

## 1.3 性能工程：突破物理极限
*   **零开销隔离切换**：域间切换等价于一次 VM-Exit 与 VM-Entry。由于每域持有独立 EPT，无需刷新 TLB，确保缓存局部性。
*   **中断解耦与直接门 (Direct Doorbell)**：利用 x2APIC 及预共享内存队列，允许不同核心的隔离域互发 IPI 并直接唤醒，消除 Hypercore 介入带来的开销。
*   **绝对硬件直通**：利用 SR-IOV，网卡/存储设备的虚拟功能 (VF) 直接接入隔离域，DMA 路径实现真正的应用直达硬件。

## 1.4 安全模型：零信任与物理强制
系统默认拒绝任何隐式共享，消除传统“超级用户/Root”概念。
*   **EPT 强制内存边界**：硬件 MMU 并行校验内存边界。即使 Guest Ring 0 被攻破，越界访问仅会触发 EPT Violation，导致该域被终结，绝不波及宿主及其他域。彻底免疫跨域 Spectre/Meltdown 漏洞。
*   **IOMMU 强制总线边界**：闭源驱动即便发起恶意 DMA 请求，也会在 PCI 总线层被硬件阻断。
*   **后量子密码学 (PQC) 建议**：架构级建议在存储快照签名和跨机网络隧道协商中，采用基于格的后量子算法（如 CRYSTALS-Kyber/Dilithium）。

## 1.5 高级韧性与实时恢复
隔离域内软件崩溃被定义为局部硬件异常，由 Hypercore 捕获并执行预设策略：
*   **无状态域闪电重启**：微服务引发异常后，Hypercore 瞬间销毁其 VMCS 与 EPT 并重建，恢复时间为微秒级。
*   **关键域一致性快照**：数据库等有状态域通过存储管理域定期生成原子快照，崩溃后回滚至极小丢失视窗。
*   **物理硬件降级预测**：Hypercore 监听 MCE / APEI 事件（如内存 ECC 错误率飙升），通过动态重写 EPT，将受影响域热迁移至安全物理页，阻断硬件故障扩散。
