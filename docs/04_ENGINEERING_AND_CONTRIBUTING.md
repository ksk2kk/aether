# 4. 架构工程规范与社区治理

Hypercore 是整个 Aether 矩阵的信任根，维护其确定性与安全边界是核心使命。系统内核逻辑与上层发行版生态在开发模式上执行极其严格的分野。

## 4.1 极简工程规范
1.  **语言基准**：采用 Rust Nightly `no_std` 模式，屏蔽所有底层运行库依赖。
2.  **隔离 Unsafe 边界**：除 `src/arch/` 体系结构 HAL 层内封装的内联汇编及硬件寄存器读写外，严禁在 `vm`、`memory`、`msb` 及调度层使用任何 `unsafe` 块。
3.  **禁止非确定性依赖**：核心代码不允许引入具有隐式状态的第三方库。并发控制统一采用 `spin::Mutex` 结合局部中断屏蔽。

## 4.2 核心目录规范
```text
src/
├── arch/x86_64/    # VMX 状态机、IOMMU 控制器、x2APIC
├── memory/         # Buddy 分配器、EPT 嵌套页表、物理到虚拟拓扑映射
├── vm/             # VM-Exit 派发器、Hypercall ABI 截取器
├── msb/            # 内存语义总线引擎及 Capability 授权表
├── mmdl/           # 全局物理去重哈希账本
├── fusion/         # 增强 POSIX 宏域的 I/O 桥接存根
├── enclave/        # VMCS 生命周期与核心调度器
└── lib.rs          # 极简的 kernel_main 引导点
```

## 4.3 任务分解结构 (WBS)
工程落地按照阶段演进：
*   **阶段 I：自举与控制权**。Multiboot2 引导，接管 VMX Root 及内存拓扑（已完成）。
*   **阶段 II：内存隔离与 EPT**。实现物理帧 Buddy 分配及四级嵌套页表（已完成）。
*   **阶段 III：异常与中断解耦**。涵盖 VM-Exit 全量框架、APIC 接管及 x2APIC 门铃通信（已完成）。
*   **阶段 IV：MSB 与 IOMMU 边界**。构建基于硬件审计的 DMA 隔离及总线通信原子操作（核心机制已达标）。
*   **阶段 V：核心服务与验证基底**。MMDL 原生持久化结构、Aether Fusion 桥接注册，及首个 Init 域扁平装载（阶段性交付点）。

## 4.4 治理模式：精英中心与生态去中心
*   **技术指导委员会 (TSC)**：由底层虚拟化与形式化方法专家组成，负责驳回任何试图向 Hypercore 引入“系统便利性”及特权调用的设计。
*   **RFC 强制要求**：所有对内核 ABI、调度策略或硬件树的重构，需提交详尽的 RFC 提案，并在可能的情况下附加 TLA+ 数学规范论证。
*   **客观 CI/CD 仲裁**：代码合并完全依赖自动化环境的 VM-Exit 风暴测验及 IOMMU 越界压力测试，排除主观偏见。
*   **去中心化生态**：基于 Hyperion 构建的上层操作系统及隔离域服务由社区自由演化，发行版只需对齐 Hypercall ABI 即可具备通用性。
